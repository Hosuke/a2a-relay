# A2A Relay v1.3 GPT-5.5 Design Review

Date: 2026-05-09
Reviewer: GPT-5.5
Status: design critique, no code changes

## Executive Verdict

The v1.3 direction is sound if it stays narrow: safe automatic replies between
two private agents, backed by local policy, metadata-only operator surfaces, and
an auditable filesystem mailbox. The current repo already has the right safety
shape for that: receipt watchers, request-only dispatch, sender allowlists,
`human_approval_required`, bounded stdout replies, `shell=False`, message
claiming, dedupe, and metadata-only `pending` / `queued` / `timeline` outputs.

The dangerous part of v1.3 is scope creep. GitHub PR/design interaction and
multi-agent Cursor orchestration introduce public-input, authority, and
coordination problems that are not solved by the current private mailbox
protocol. They should be designed now, but only the lowest-risk bridge pieces
should ship in v1.3.

Recommendation: ship v1.3 as a hardened private two-agent auto-reply and task
handoff release. Add protocol fields and tests that make future GitHub and
multi-agent orchestration possible, but postpone public GitHub automation,
multi-agent write orchestration, group chat, and arbitrary agent-to-agent task
routing until the loop and policy invariants are mechanically enforced.

## Current Baseline

What is already strong:

- Message bodies are data, not commands.
- Dispatch only runs pre-registered local `argv` with `shell=False`.
- `reply`, `status`, and `heartbeat` are never dispatched.
- `human_approval_required=true` queues instead of running.
- `receipt` defaults to queueing `request` messages.
- CLI operator views intentionally omit message bodies.
- Event logs record state transitions with message/thread metadata.
- Atomic claim into `processing/<agent>/` reduces double processing.
- Seen state dedupes by `id` and optional `idempotency_key`.
- Contacts and dispatcher policy are local files, not message-controlled.

Important gaps relative to a v1.3 safety bar:

- The roadmap mentions `hop_count` / `max_hops`, but the schema and dispatcher
  do not enforce loop depth today.
- Auto-generated replies are not explicitly marked as such.
- There is no per-thread auto-turn budget or "one automatic reply per inbound
  request" invariant.
- Dispatcher policy is split between CLI arguments and `dispatcher.json`.
  Some documented policy fields, such as `max_body_chars` and `allowed_types`,
  are not the single source of truth for dispatch eligibility.
- Signing fields are reserved but not enforced. This is acceptable for a trusted
  private filesystem mailbox, but not for GitHub/webhook/public input.
- Dispatch failure currently logs failure and sends no reply/status. v1.3 must
  make sender-visible failure semantics explicit.
- Audit events are metadata-only, which is good for privacy, but there is no
  event-level policy decision record with the exact gate inputs and outcome.

## Protocol Safety

v1.3 needs a tighter protocol contract before expanding automation.

Required protocol additions for v1.3:

- `auto_generated: boolean`
  - Set to `true` on dispatcher-produced replies and statuses.
  - Auto-generated messages must default to `needs_reply=false`.
- `causal_chain: string[]` or `root_message_id`
  - Preserve a compact chain from original request to reply/status.
  - This gives loop detection more than `thread_id`.
- `hop_count: number` and `max_hops: number`
  - Increment on every automated outbound message.
  - Reject or queue when `hop_count >= max_hops`.
  - For v1.3 private auto-reply, default `max_hops=1`.
- `policy_profile: string`
  - Names the receiver-local policy that authorized processing.
  - The sender may request a profile in the body or capabilities, but the
    receiver chooses and records the enforced profile.
- `capabilities_granted: string[]`
  - Distinguish requested capabilities from granted local capabilities.
  - Replies should record what was actually allowed.
- `body_sha256: string` in events, not necessarily in the message.
  - Allows audit correlation without exposing full bodies in timelines.

Fields that can remain reserved/postponed:

- Attachments and binary payloads.
- Group/thread membership.
- WebSocket session metadata.
- Full public-key signing if v1.3 remains trusted filesystem only.

## No-Loop Guarantees

The current "never dispatch replies/status/heartbeat" rule is necessary but not
sufficient once more automation exists.

v1.3 should define these invariants:

1. Only `type=request`, `needs_reply=true`, `human_approval_required=false`, and
   allowed sender messages are eligible for automatic dispatch.
2. A dispatcher-generated `reply` must set `needs_reply=false`.
3. A dispatcher-generated `status` must never be treated as a request.
4. A message with `auto_generated=true` must not trigger another automatic
   dispatch unless a receiver policy explicitly allows that exact message type
   and the hop budget has remaining capacity.
5. Each `(message_id or idempotency_key, action_name)` can run at most once.
6. Each thread has a bounded auto-turn budget, defaulting to one automatic
   receiver turn in v1.3.
7. Human approval resets nothing automatically; it creates a new explicit human
   decision event.

The minimum v1.3 loop-proof shape is: private request in, at most one automatic
reply or failure status out, no automatic processing of that reply/status.

## Privacy And Leak Risks

The repo is already careful not to echo message bodies in common CLI summaries.
That should remain a hard requirement.

Remaining privacy risks:

- Full bodies are still stored in inbox, processing, and archive JSON. That is
  expected for an auditable mailbox, but it needs retention and permissions
  guidance.
- Dispatcher stdout becomes reply body. A local agent can accidentally return
  secrets, environment values, paths, private repo diffs, or credentials.
- GitHub PR/design integration would expose untrusted public or semi-public text
  to local agents and could cause private context to be posted back to GitHub.
- Multi-agent Cursor orchestration may leak one agent's prompt/context into
  another agent's request or reply unless runs have explicit boundaries.

v1.3 requirements:

- Keep operator summaries metadata-only.
- Add a redaction/check hook before auto replies are sent.
- Add a configurable denylist for obvious secret patterns.
- Record reply body length and hash in events, not the body.
- Document that archive files contain full message bodies and must be protected.
- Do not bridge private mailbox contents into GitHub comments without an
  explicit policy gate and human approval.

Postpone:

- Automatic posting of agent-generated replies to public GitHub PR comments.
- Cross-thread or cross-agent memory sharing.
- Attachments or copied repo files inside messages.

## Auditability

Current JSONL events are a good foundation. v1.3 should make audit records
decision-oriented, not just lifecycle-oriented.

Add a policy decision event for every claimed message:

- `event_type`: `policy_decision`
- `message_id`, `thread_id`, `actor`
- `decision`: `dispatch`, `queue`, `archive`, `reject`
- `reason`
- `policy_profile`
- `action`
- `capabilities_granted`
- `auto_generated`
- `hop_count`
- `body_sha256`
- `reply_message_id` when applicable

This event should be written even when a message is skipped. Auditors should be
able to answer: who sent this, why was it or was it not run, what local action
ran, what reply was produced, and which policy allowed it.

For GitHub and Cursor orchestration, also record:

- External source identifiers: repo, PR number, comment id, event id.
- Local run identifiers: agent id, action id, working directory or worktree,
  branch, commit range, and whether writes were allowed.
- Human approval references when a gate is overridden.

## Policy Gates

The v1.3 policy model should be local, explicit, and fail closed.

Required gates:

- Sender is a known private contact.
- Recipient matches the local agent.
- Transport trust level matches the policy.
- Message type is allowed by the active policy.
- `needs_reply` is required for auto-reply.
- `human_approval_required` blocks automation.
- Body size is enforced from the same policy source that selects the action.
- Capabilities requested are mapped to capabilities granted.
- Action is pre-registered and not selected by the message body.
- Working directory, timeout, stdout limit, and environment are fixed by policy.
- Auto-generated messages obey `hop_count`, `max_hops`, and per-thread budgets.
- Failed dispatch behavior is explicit: either metadata-only failure status or
  no sender-visible status, but not an accidental mix.

Recommended policy cleanup:

- Make `dispatcher.json` the single source for dispatch policy.
- Treat CLI `--allow-from` as an additional narrowing filter, not the primary
  policy for dispatch.
- Validate `dispatcher.json` at startup and refuse ambiguous or partial policy.
- Add policy versioning so event logs can reference the exact policy shape.

## GitHub PR And Design Interaction

GitHub should be treated as an untrusted external transport, even for private
repos. PR titles, bodies, comments, commit messages, and design text can contain
prompt injection and malicious instructions.

Safe v1.3 scope:

- Read-only GitHub design review ingestion into the mailbox.
- Metadata-only events linking a mailbox thread to a GitHub PR/comment.
- Human-approved draft replies, not automatic posting.
- A narrow command grammar for maintainers, such as "draft review" or "summarize
  CI", with repo and actor allowlists.

Must not be in v1.3:

- Automatic code changes from arbitrary PR comments.
- Automatic posting to GitHub without human approval.
- Public webhook ingestion without signature and replay verification.
- Any flow where GitHub text selects a local command or Cursor profile.

Required before broader GitHub automation:

- GitHub App or webhook signature verification.
- Delivery id dedupe and replay protection.
- Repo/owner/branch allowlists.
- Actor permission checks.
- Rate limits and comment storm protection.
- Clear separation of public PR content from private local context.
- CI/status gate before merge-related actions.

## Multi-Agent Cursor Orchestration

Multi-agent Cursor orchestration is powerful but should not be modeled as chat.
It is closer to a scheduler assigning isolated runs with explicit authority.

Safe v1.3 scope:

- One coordinator can send a private task request to one worker agent.
- Worker starts in receipt-only or draft-only mode.
- Read-only local investigation can be auto-run under a named profile after
  receipt watcher stability is proven.
- The reply includes evidence, commands run, and whether approval is needed.

Must not be in v1.3:

- Multiple agents editing the same checkout without worktree isolation.
- Agents delegating recursively to other agents.
- Any agent modifying another agent's policy/config.
- Automatic merges, pushes, force pushes, migrations, restarts, or deletes.
- Shared memory writes accepted from messages.

Required orchestration controls before expansion:

- Run id, lease/claim id, and cancellation semantics.
- Worktree or branch ownership per agent run.
- File/path ownership or conflict policy.
- Tool permission profile recorded in audit events.
- Maximum concurrent agents per repo/thread.
- Human approval for writes outside the assigned worktree or profile.

## What Must Be In v1.3

Ship these as the v1.3 core:

- Hardened private two-agent automatic reply.
- Receipt watcher stability and recovery.
- Explicit no-loop fields and enforcement.
- `auto_generated`, `hop_count`, `max_hops`, and causal/root message tracking.
- Dispatcher policy validation and single-source policy enforcement.
- Metadata-only audit events with policy decisions and body hashes.
- Failure semantics for dispatch.
- Redaction/check hook before automatic replies are sent.
- Focused tests proving no reply/status/heartbeat loop, duplicate dispatch
  prevention, human approval queueing, and body privacy in summaries.
- Documentation that v1.3 is private-contact automation, not group chat or
  public webhook automation.

## What Should Be Postponed

Postpone these until after v1.3:

- Public GitHub webhook automation.
- Automatic GitHub comment posting.
- GitHub-triggered code edits.
- Multi-agent write orchestration.
- Recursive delegation.
- Group chat.
- Attachments and file transfer.
- WebSocket transport.
- Shared memory acceptance.
- Dynamic policy changes through messages.
- Generic plugin/action loading.

## Test Plan

Protocol and validation:

- Reject missing or invalid `hop_count`, `max_hops`, and `auto_generated`.
- Reject or queue when `hop_count >= max_hops`.
- Verify dispatcher replies set `auto_generated=true` and `needs_reply=false`.
- Verify causal/root id is preserved across request, status, and reply.
- Verify oversized bodies use policy-configured limits.

Loop safety:

- Auto-generated reply in receiver inbox does not dispatch.
- Status ACK does not trigger ACK recursion.
- Duplicate `id` and duplicate `idempotency_key` do not re-run actions.
- Same request cannot run the same action twice after restart.
- Thread auto-turn budget blocks a second automatic turn.

Policy gates:

- Unknown sender, ambiguous alias, wrong recipient, disallowed type, empty
  allowlist, disabled policy, and `human_approval_required=true` all fail closed.
- Message body cannot select action, cwd, timeout, profile, or environment.
- CLI narrowing cannot broaden `dispatcher.json` policy.
- Invalid dispatcher config fails before processing messages.

Privacy:

- `pending`, `queued`, `threads`, `timeline`, dispatch output, and receipt output
  never include message bodies or secret markers.
- Event logs include body hash/length but not body.
- stderr is never included in replies.
- Redaction hook blocks or queues replies containing known secret patterns.

Auditability:

- Every claimed message emits exactly one `policy_decision` event.
- Dispatch success links original message id to reply message id.
- Dispatch failure records action, reason, and whether sender-visible status was
  sent.
- Human approval queue events preserve path and reason without body leakage.

GitHub bridge, if any read-only pieces land:

- Webhook/event id dedupe.
- Repo and actor allowlist.
- PR/comment body treated as untrusted text.
- No automatic GitHub post without explicit approval.
- Private mailbox body is never copied to GitHub by default.

Multi-agent orchestration, if any draft-only pieces land:

- Worker receives task and queues by default.
- Draft-only action cannot write files or run unrestricted tools.
- Run id and policy profile appear in audit events.
- Concurrent runs cannot claim the same task.
- Approval-required tasks stay in `processing/`.

## Final Recommendation

The v1.3 release should be marketed and engineered as "private, bounded,
auditable auto-reply and task handoff." That is valuable and reachable from the
current codebase.

GitHub PR/design interaction and multi-agent Cursor orchestration should remain
behind design flags unless they are read-only or draft-only. The protocol should
grow the safety fields now, because adding them after public/webhook or
multi-agent write automation would be much harder and riskier.
